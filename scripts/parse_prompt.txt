## Extraction Instructions

Your job is NOT to produce a generic summary.
Your job is to transform each paper into reusable knowledge that can answer real user questions later.

The stored knowledge must help answer questions such as:
- What problem does this method solve?
- When should this method be used?
- What are the assumptions, inputs, outputs, and limitations?
- How does the method work in practice?
- What do the parameters mean?
- What evidence supports the claims?
- How is this method different from related methods?
- What can fail, and under what conditions?

Do NOT store vague, generic, motivational, or high-level filler statements.
Do NOT store canned summary sentences unless they are truly useful for future Q&A.
Optimize for future retrievability, factual precision, and usefulness in answering realistic technical questions.

For each paper, extract and store knowledge following these rules:

1. Preserve concrete technical content:
   - definitions, assumptions, formulas or algorithmic steps
   - variable meanings, parameter roles, input/output behavior
   - edge cases, experimental findings, limitations
   - implementation details, failure modes

2. Break knowledge into small self-contained units.
   Each unit should be understandable without needing the whole document.

3. Prefer Q&A-useful statements over narrative summary.
   Bad: "This paper proposes an interesting and effective method."
   Good: "The method improves feature recovery under nonlinear indicator-type signals by expanding candidate paths and then trimming redundant ones."

4. When processing papers, extract:
   - research question
   - method core idea
   - mathematical assumptions
   - identification conditions / theoretical guarantees (include specific convergence rates like O(n^{-1/2}))
   - comparison baselines
   - experimental settings
   - what conclusions are actually supported by results
   - what is NOT supported

5. Explicitly capture uncertainty.
   If the source is ambiguous, incomplete, or speculative, mark it clearly.

6. The field "reusable_for_questions" must contain realistic future questions this knowledge can help answer.

7. Prefer granularity over merging.
   If two passages discuss different aspects (e.g., definition vs. limitation vs. convergence rate), keep them as SEPARATE units.

8. Every stored unit must pass: "Would this help answer a real technical user question later?" If no, do not store it.

9. Target: Extract 8-25 knowledge units per paper. If you produce fewer than 8 for a full paper, you are likely being too aggressive in merging.

## Output JSON Schema

For each paper, save a JSON file containing an object with:
- "paper_metadata": the paper's metadata (title, authors, year, doi, domain, cluster, filename)
- "knowledge_units": an array of knowledge unit objects

Each knowledge unit must have exactly these fields:

```json
{
  "source_type": "paper",
  "title": "method name (NOT paper title)",
  "section": "which section this came from (e.g., 'Section 2: Methodology', 'Abstract')",
  "knowledge_type": "definition | assumption | algorithm | parameter | result | limitation | implementation | failure_mode | comparison",
  "topic_tags": ["tag1", "tag2"],
  "question_intent_tags": ["what_is_it", "when_to_use", "how_it_works", "why_failed", "compare_methods"],
  "content": "self-contained knowledge statement",
  "evidence_span": "original passage, formula, or line reference from the paper",
  "dependencies": ["related symbols, functions, prior concepts"],
  "limitations": "what this statement does not imply",
  "confidence": "high | medium | low",
  "reusable_for_questions": ["Example question 1?", "Example question 2?"],
  "method_name": "canonical name of the method (e.g., OGA, DML, Deep Kriging)",
  "field": "statistical sub-field (e.g., high-dimensional regression, causal inference)",
  "keywords": ["keyword1", "keyword2", "keyword3"],
  "problem_it_solves": "one sentence: what practical problem does this method address?",
  "model_assumption": "key assumptions the method requires (e.g., sparsity, linearity)",
  "input_format": "what data/inputs the method expects",
  "output_format": "what the method produces",
  "typical_questions": ["A realistic question a researcher would ask about this method"],
  "related_methods": ["method1", "method2"]
}
```

Field rules:
- "title" is the METHOD name, NOT the paper title
- "section" is the specific section of the paper, NOT the paper title
- "method_name": ALWAYS use the PRIMARY method the paper proposes. Sub-concepts go in "related_methods"
- "field": Map to a recognized statistical sub-field, not a vague category
- "keywords": Include both formal terms and informal synonyms researchers might search for
- "problem_it_solves": Be specific. Not "solves regression" but "selects relevant variables when p >> n"
- "model_assumption": List concrete mathematical or data assumptions
- "input_format" / "output_format": Be specific about data structures, dimensions, types
- "typical_questions": Write 1-3 questions a real researcher would ask, in casual or applied language
- "related_methods": Name specific alternative or complementary methods

## Workflow for Each Paper

1. Read the PDF file using the Read tool
2. Extract knowledge units following the rules above
3. Write the output JSON file using the Write tool
4. Move to the next paper

Return ONLY valid JSON in each output file, no markdown fences, no extra text.
